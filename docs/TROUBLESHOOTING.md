# トラブル対策まとめ — 実際にハマった所と解決策

このAIアシスタント（および姉妹プロジェクト RET / EAST）を作る過程で**実際に発生した不具合**と、その**原因・解決策・再発防止**を記録したものです。
同じ仕組みを作る人が同じ穴に落ちないための実戦ノートです。

> 読み方: 各項目は「症状 → 原因 → 解決 → 教訓」で統一しています。AIに丸ごと読ませて「同じ轍を踏まないように実装して」と指示するのにも使えます。

---

## 1. 日本語RAGで「知ってるはずのこと」に答えない

**症状**: Obsidianに「AIニュース」「クロード」のノートがあるのに、聞いても「記憶にありません」と返る。

**原因**:
- 検索が**本文(chunk)だけ**を対象にしており、**ファイルパス/タイトル**にしか無いキーワードを取りこぼしていた。
- 質問が「クロード」（カナ）なのに、ノート本文は「Claude」（英語）。**表記揺れ**で一致しない。

**解決**:
- 検索対象に `path`（パス/タイトル）も追加。`WHERE chunk LIKE ? OR path LIKE ?`
- カナ→英語の**シノニム辞書**を用意（`クロード→claude`、`ジェミニ→gemini` 等）して検索語を拡張。

```js
const KEYWORD_SYNONYMS = { 'クロード':'claude', 'ジェミニ':'gemini', 'エーアイ':'ai', ... };
```

**教訓**: 日本語RAGは「本文一致」だけでは足りない。**パス/タイトル＋表記揺れ吸収**が再現率(recall)の生命線。

## 2. 短い単語が拾われ過ぎる（精度の崩壊）

**症状**: 「AIについて」で検索すると、英単語に "ai" を含むノートが大量ヒット（例: 319件中147件）して、関係ない記憶まで引っ張る。

**原因**: 2文字の汎用語（"ai" など）が英文中のあちこちにマッチしてしまう。

**解決**: **パス/タイトル一致を本文一致の×3で重み付け**してスコアリング。タイトルにキーワードを含むノートを優先し、本文のたまたま一致はノイズとして沈める。

```js
e.score += path.includes(term) ? 3 : 1;   // パス一致は3倍
```

**教訓**: 再現率(recall)を上げると精度(precision)が落ちる。**重み付けスコアリング**で両立させる。「アバウトな質問にも引っかかる」と「ノイズを出さない」はトレードオフ。

## 3. Gitに知らない人が「コントリビューター」として記録される

**症状**: GitHubにpushしたら、関わっていない第三者（同僚）の名前がコントリビューターに出た。

**原因**: ローカルの `git config user.email` が、過去に使った別人のメールアドレスのままだった。コミットの著者はこの設定値で決まる。

**解決**:
- コミット前に必ず著者を確認・固定:
  ```bash
  git config user.name  "あなたの名前"
  git config user.email "あなたのnoreplyメール"   # 例: 12345+user@users.noreply.github.com
  ```
- 既に混入した履歴は `git filter-branch`（または `git filter-repo`）で書き換え。

**教訓**: **公開前に著者を必ず確認**。特に複数アカウント/他人のマシンを触った後は要注意。AI協働の共著表記(`Co-Authored-By`)はOKだが、無関係な第三者を著者にしない。

## 4. 公開リポに「パスワードが埋め込まれていた」

**症状**: ポートフォリオとして公開しようとしたら、コードに固定パスワード（数字だけの固定値）がハードコードされていた。しかも**初回コミット履歴にも残存**。

**原因**: 動くものを優先してパスワードを直書きしていた。ファイルを消しても**git履歴には残る**。

**解決**:
- 公開用は**別リポジトリを新規作成**（履歴ごと作り直し）。秘密情報が一度も入らない状態に。
- 秘密は必ず**環境変数/シークレット**へ（`wrangler pages secret put` 等）。コードには置かない。
- 公開前に `grep -rniE "password|token|api[_-]?key|sk-|AIza"` でスキャン。

**教訓**: **公開＝取り消し不可**（キャッシュ・インデックスされる）。公開前に①秘密情報スキャン②履歴の点検③不要ファイル削除。実データ・本番資格情報を含む本体はPrivateのまま、**仕組みだけ複製した公開用リポ**を作るのが安全。

## 5. 無料枠の上限で突然止まる

**症状**: 一定回数使うとAIが応答しなくなる（EASTで Workers AI のNeuron日次上限に到達）。

**原因**: クラウドの**無料枠レート制限**。大きめのモデルを多用すると早く枯れる。

**解決**:
- 用途で**LLMを使い分け**る（重い推論=ローカルLlama/Ollama、対話=API など）。
- このアシスタントでは生成を **Gemini API（gemini-2.5-flash）** に集約し、軽量・安定運用。

**教訓**: 無料枠は「いつか必ず尽きる」前提で設計。**フォールバック先**とモデル選択をあらかじめ決めておく。

## 6. AIの出力が英語になってしまう（日本語TRPG）

**症状**: 日本語で使いたいのに、モデルが英語で返答する。

**原因**: 使ったモデル（llama3）が日本語指示に弱く英語へ流れた。

**解決**: 日本語が得意なモデルへ変更（例: `gemma2:2b`）。Gemini系は日本語が安定。

**教訓**: 「動く」と「日本語で自然」は別問題。**言語適性でモデルを選ぶ**。プロンプトに「必ず日本語で」と明記するのも有効。

## 7. 非同期の状態レースで処理が途中で死ぬ

**症状**: 一連のシーケンス（音声認識→画面遷移→次の動画）が、途中で待機状態に戻ってしまう。

**原因**: 音声認識の `onend` が結果確定**後**に発火し、状態を IDLE に戻すハンドラがシーケンスを打ち切っていた。**イベントの発火順とstate遷移の競合**。

**解決**: シーケンス実行中は `_seqLock` フラグを立て、`onend` 等の割り込みが state をリセットしないようガード。

**教訓**: イベント駆動UIでは**「今、意図したシーケンスの最中か」を明示的に持つ**。コールバックの発火順を仮定しない。

---

## 8. ファイル書き込みが「すでに存在する」で失敗

**症状**: 新規作成のつもりが、既存ファイル（テンプレからのコピー）の上書きで弾かれた。

**原因**: 流用元をコピー済みだったのに新規扱いしていた。

**解決**: 上書き前に必ず**現状を確認（Read）**してから編集/書き込み。

**教訓**: 「新規のはず」を疑う。**触る前に現物を見る**。

---

## 9. 「Macが起動していないと使えないのでは?」問題

**症状**: 自宅PCに依存していると、外出中・PCオフでアシスタントが使えない。

**原因**: 初期設計でリクエスト経路に自宅マシンを含めていた（`User→CF→Mac→LLM`）。

**解決**: **クラウド完結アーキテクチャ**へ。
- 記憶は Obsidian → **D1（クラウド）にミラー**。
- 読む・答える・推論は **Cloudflare + Gemini** で完結。
- Macが要るのは「**新しいノートを取り込む同期**」のときだけ。

**教訓**: パーソナルAIは「**読む(クラウド完結) / 取り込む(同期時だけローカル)**」を分離すると、可用性が劇的に上がる。同期は**冪等**に（重複実行しても壊れない）。
